    What, who and why is INetD?
    First of all, I hope you know what a daemon is. If not, read out tutorials here.
    Now, let's assume that you understand what a daemon is. So anyway, daemons run as TSRs. TSR stands for "Terminate and Stay Resident". A TSR, once executed, will stay on your system's memory and wait for a certain action to take place (in our case: a connection on a certain port), and then do something about it.
    Now, common sense says that a machine that is running multiple daemons will need lots of system resources for that. This is where INetD comes to place.
    INetD stands for Internet (I-Net) Daemon (D). It is often called "the mother of all daemons". Why? Because it is a single TSR that is in charge of invoking other daemons.
    Here, let me quote from the book Maximum Security (read it online or buy it online )

    Common sense tells you that running a dozen or more daemon processes could eat up machine resources. So rather than do that, why not create one
    daemon that could listen for all the others? That is what inetd does. It listens for connection requests from the void. When it receives such a request, it
    evaluates it. This evaluation seeks to determine one thing only: What service does the requesting machine want? For example, does it want FTP? If so,
    inetd starts the FTP server process. The FTP server can then process the request from the void. At that point, a file transfer can begin. This all happens
    within the space of a second or so.


       TIP: inetd isn't just for UNIX anymore. For example, Hummingbird Communications has developed (as part of its Exceed 5 product line) a
         version of inetd for use on any platform that runs Microsoft Windows or OS/2. There are also non- commercial versions of inetd, written by
         students and other software enthusiasts. One such distribution is available from TFS software and can be found at
         http://www.trumpton.demon.co.uk/software/inetd.html.

    The SU command
    SU is a very useful Unix command. SU stands for Switch User. SU let's you run a sub-shell with the priviledges of another user within your current shell.
    Basic usage: suppose you know someone else's password, whether he gave it to you or you hacked his account. Simply type "su user password" (without the quotes), and replace user with this person's username, and replace password with his/her password.
    For root access: if no account is provided (just su and a single parameter. For example: "su blah"), your system assumes that you want to su to root, and checks if this is root password. If you entered the correct password, you get root access (unless the admin played around with the password file and gave root false permissions. In such a case, simply find the password file, find a user who has UID 0 and GID 0 and su to it (assuming that you have his/her password, of course. If not, try some common passwords)).
    When su'ing from root: remember that if you have root priviledges, nothing is supposed to stop you. So why would you need to have someone's password just to switch to his/her user? Simply type "su username" (without the quotes) to login as that user.
    When su'ing to a passwordless user: type "su username -" (without the quotes).
    To exit the sub-shell: simply type "exit" (without the quotes) to exit the sub-shell and return to your normal shell.
    How the hell can su read the shadow file?: first of all, if you don't know what the shadow file is, read this. Now, su is an SUID-root program. This is why it can read the shadow file, which only users with root access can read. To learn about SUID files, click here.

    A nifty usage : most systems will block remote root logins, so people won't be able to get root even if they know root's password. But, if you know someone else's password, you could login as this user, get a shell and type "su root's-password" (without the quotes, of course).
     

    What is SUID?
    SUID stands for Set User ID. Here, let me explain.
    Now, as you should know by now, every program or command you execute runs with YOUR priviledges. Right? WRONG!
    SUID files can be run with the priviledges of the person who SUIDed them. You see, when you SUID a program, it puts your user ID somewhere inside the program so the system will know to run it with the SUIDer's priviledges. Of course, if you have root access, this should not trouble you, because you already have maximum priviledges.
    SUID programs also pose a great security threat, especially SUID-root programs. If someone finds a way to make these programs do something they shouldn't do, he can do it with root priviledges. For example: if you could make an SUID-root program execute a command for you, you could tell it to execute sh to run a Bourne Shell (or BASH for Bourne Again Shell, CSH for C Shell, TCSH for Turbo C Shell, KSH for Korn Shell etc') with root priviledges (or in other words, a root shell).
    Sometimes, on the other hand, you have to use SUID programs (such as SU, for example), so make sure you don't screw up with your security.

    Want to learn how to SUID files? Read chmod's man page by typing 'man chmod'.

    What are Linux runlevels?
    If you've been paying attention to what your Linux box does during startup, you should have noticed that it says: "Entering runlevel x" (where x is a number between 1 and 5) at one point of the bootup stage (after it mounts your root filesystem (your "/" directory) into read-write mode, sets up sound, finds your kernel's module dependencies (never mind that for now) etc'). A runlevel is a bootup/shutdown sequence. It consists of a list of commands to run on startup and a list of commands to run on shutdown (or when switching to different runlevels).
    Now, first of all, let's see how you can switch runlevels. Bah, that's easy. Simply type init x, where x is a number between 0 and 6. Runlevel 0 is for "halt" (turning off your computer, if you have APM -Advanced Power Management, and if you have APMD - APM Daemon, installed. All modern CPUs have APM), runlevel 6 is for reboot and the rest are various runlevels. 5 will bootup everything - it will even automatically run X (by default, of course. You can change this). Runlevel 1 is considered the single-mode runlevel. It does the least possible (kinda like "safe mode" in Windows) and doesn't even require you to enter a password (but only root can switch runlevels, so you have to be either root or have physical access to the computer during startup (we'll get to that later)).
    Editing your runlevels list is different with every different distribution. Usually it would go like this:
    a) Go to /etc/rc.d/rcx.d/ (where x is the runlevel's number) and play around within this directory. It contains symbolic links (kinda like shortcuts in Windows. For more information about symbolic links (otherwise known as "symlinks"), type man ln) to programs (including their parameters) that will be executed, and symbolic links to programs that will be killed on shutdown. Play around to find out more (but ALWAYS make backups!!).
    You can also try this:
    b) (this should work on most Linux boxes) Switch to the runlevel you want to edit. Then type setup. Go to system services, and select/unselect the services u want to run on startup and kill on shutdown).
    Or this:
    c) The easiest way - on some distributions, such as RedHat, you will be able to type the command control-panel within an xterm (a "virtual terminal" - a console window within X-Windows) and get a nice little window thingi with lots of buttons and suchlikes. Find the button that says "runlevel editor" when you put the mouse above it for a second or two. Then click on this button and play around with the programs. I'm sure you'll figure out how to use it yourself. It's quite self-explanitory, and it contains help files and documentation if you really need help.
    Anyway, you can always try running the command:
    find / -name *rc* -print
    If none of the three specified methods didn't work, so you could see where your runlevel directories are and apply method a on them.

    And now, for a nice little runlevels-related hack.
    Now, if you're reading this document, you're probably a Linux newbie, so you probably use Redhat Linux, Mandrake Linux etc'. If so, you should have a prompt saying "boot:" or "LILO boot:" or "LILO:" when you start your computer, and you could either type Windows or Linux (you can change these names into, say, sucky-OS for Windows and Stable_and_secure_OS for Linux, or anything else you want. Use the linuxconf program to edit LILO's preferences, and use your imagination...  :-) ). Now, what happens if you type linux 5? Of course! It boots up Linux in runlevel 5!! But wait! What happens if you type linux 1 or linux single? It runs on runlevel 1 - single user mode, which means... automatic root access! No password needed.   :-)
    Of course, you can change runlevel 1 by going to /etc/rc.d/rc1.d and then removing the contents of this directory and copying everything from another runlevel (replace the number in rc1.d to any other runlevel number). Please read cp's manual page and make sure you're copying symbolic links as links and not as files.
    Of course, anyway, if you have local access to a Unix box (or any other box that I know of, as a matter of fact), you can boot from a floppy (unless, of course, the first boot device is a hard drive and the bios is password-protected, but these can be hacked too. Refer to this guide).
     

    What are Winmodems, and why do they suck? + Getting online with a WinModem from Unix/Linux
    First of all, what is a MODEM anyway? The initials modem stand for modulator/demodulator. To make a long story short, a modem translates digital signals which your computer sends it into signals that can be transferred through the phone line (if it's a cable modem or some other kind of modem, it'll translate computer signals into whatever signals that can be transferred through the communications line it is supposed to run on), and then receives back phone signals and translates them into bits and bytes your computer can read and understand.
    Note: of course, your modem does more than that. For example: it provides the actual connection with the phone line, and quite a variety of other things which are currently unimportant to us.
    This whole modulating (turning computer signals into phone signals) and demodulating (duh... the opposite of modulating. Turning phone signals into computer signals) process requires some software, and of course, a computer to run it. So your modem actually contains a little computer that does the modulating and demodulating.
    Then what is a Winmodem? A Winmodem is a modem that does not do the modulating and demodulating thing. Instead, it leaves it up to your computer to do that. It comes with the software necessary for your computer to perform these convertions.
    So why do they suck so much? Because:
    a) This whole convertion process takes up some resources from your computer, such as memory and CPU time.
    b) If your computer slows down for some reason, you might get disconnected all of a sudden.
    c) They don't run on any version of Unix. Why? Simply because there is no Winmodem software for Unix. The vendor didn't bother making them, I assume (silly bunch of lamers...  :-) ).
    So why would a person want to buy a Winmodem? Simply because they cost less (I guess you could figure out why at this point).

    Later addition: You can now go online from Linux with a WinModem! Check out http://www.linuxdoc.org/HOWTO/Winmodems-and-Linux-HOWTO.html for more information! Thanks to Striker-Man for pointing that out.
     

    How do I recreate files which I have already removed from the recycle bin in Windows?
    Once upon a time, in times when people either used versions of Windows prior to 95 or didn't use Windows at all (see the history of DOS paper), there was a utility called undelete. It served as a recovery tool for deleted files that were deleted less than a week ago, and used to save a lot of work and tears. That was all nice and smooth, until Windows 95 came up with it's "recycle bin".
    Wow, looky here! Whenever I delete a file, it goes straight to the recycle bin! Ooh, I'd bet that's new...
    The recycle bin isn't some magical bin thing, it's just a hidden directory. On every Windows 95 or above hard-drive, you have a directory called recycled. Whenever you send a file to the recycle bin, it is actually moved to this directory. Sophisticated, isn't it?
    But here's the problem: most people don't like the recycle bin. They either press shift+delete to delete files without transferring them through the recycle bin or empty the recycle bin right after they send a file to it. But then, when you realize that you've made a mistake, you cannot recreate the file. Or is it so? ...
    You can use DOS's undelete to recreate deleted files that were removed less than a week ago. If the file was on the recycle bin prior to it's "final and unrecoverable deletion" (yeah, right), you will find it in your recycled directory. Can't get any easier than this. Where to find the undelete.exe program? If its not in your Windows\Command directory then you will have to copy it from your Windows CD-ROM. Its in the directory called: oldmsdos.

    This trick does not work with FAT32 pratitions. Thanks to Mortar_G for this information.
     

    Why can't I run IRC bots on a free shell, and what are IRC bots anyway?
    IRC bots can be very useful when running an IRC channel. They can greet each coming visitor, keep the channel clean and kick people who swear others, serve as a file server that is operated using commands sent in private messages and sends/receives files in DCC file transfers, and much more. IRC bots can also be used in channel takeovers, as a bot is faster than a human being, and can therefore quickly take advantage of any possible holes or opportunities that may occur (the bot would usually succeed, unless the channel operators have faster bots).
    But what is an IRC bot anyway? An IRC bot is a program, usually programmed in TCL, that was programmed to understand the IRC protocol (the same language that the IRC server and your IRC clients, such as mIRC and BitchX know and can communicate with) and react to certain situations. Let's take the simplest example of all: a greeting bot. Whenever someone joins the channel, the bot greets him by saying "Hello <nick of new user>". Bots are usually based on Eggdrop. If you want to learn more about how to make Eggdrop bots, consult our Eggdrop bots tutorial.
    To run a bot, you will need to host it on a computer that:
    a) Can run TCL. Unix and all of it's variants can run TCL using a special library, that is often pre-installed on the system already. Other systems can run TCL, but only after a lot of effort in finding the right extenral programs (have you ever noticed how many utilities come with a single average Unix/Linux distribution? Now, have you ever noticed how little come with Windows? MS Paint? Bah).
    b) Can run, operate and be online 24 hours a day, 7 days a week, so your bot will stay up all the time as well.
    Wait, wait, wait! Still don't know what a shell account is? Consult our shell accounts tutorial.
    Sounds like you're *not* looking for a free shell, like the ones nether.net gives when you telnet in and login as newuser. Free shell accounts will kill all of your processes once you log off, and will usually not let you have access to the TCL library.

    Downloading, Compiling, Installing

    Issue #66: 05/17/99 (Archive)
     

    Most Linux distributions these days use some sort of package system to manage software. For
    example, Red Hat, S.u.S.E. and Caldera's OpenLinux all use the RPM package manager to install and
    uninstall software. Debian Linux and Slackware also have their own package
    management. In an ideal world, any piece of software you wanted to run on your
    Linux box would be available as a package for your distribution.

    Unfortunately, this isn't the ideal world, and many users find themselves wanting to
    download and use software which is distributed as source code, usually in a file
    ending with the .tar.gz extension.

    Looking for instructions? They're in this document. Beware: because of the nature of the task at hand,
    these steps aren't as specific as they really need to be for most newbies, nor are they easy to follow as
    they're given. This process is not for the faint of heart!

    You should already know how to download via Netscape or ftp. Be sure to understand where you've
    saved your downloaded file and also how to visit that directory; if you're not able to do this yet, spend
    some time learning about the Unix directory structure from your favorite Unix beginner's reader. If
    you're having trouble already, you're better off doing some more study before trying to compile your
    own software; the instructions here are basic, but newbies will still find themselves stretching to follow
    along.

    What is .tar.gz? How about .tgz? And .tar.bz2?

    The .tar.gz format is similar in concept to the .zip format common in the Windows world. The .zip format
    performs two functions: it allows many different files to be rolled into one single file for convenience, and
    it compresses this single file for faster downloading (or space savings). In Linux, these two tasks are
    performed by two separate commands: the tar command and the gzip command.

    Files which end in the .tar.gz extension are the result of a two-step process. First, many files are rolled
    into a single file using the tar (Tape ARchive) program, giving the resulting file a .tar extension. Next, the
    .tar file is compressed using the gzip program, adding an additional .gz extension to the end of the file.
    Sometimes .tar.gz is abbreviated as .tgz to allow such files to exist on MS-DOS systems, which allow
    only for a single, three character filename extension. In any case, .tar.gz or .tgz files are known as
    "tarred-gzipped" files.

    Files which end in the .tar.bz2 extension are similar to the .tar.gz files in that they are multiple files which
    have been run through tar into a single file and then compressed. Files ending in .tar.bz2, however, are
    compressed with the bzip2 compressor rather than the gzip compressor. To see if your Linux system
    contains a bzip2 compressor/decompressor, type "bunzip2 --help" at the command line. If you see a
    25-line usage message, the utility is present on your system.

    How do I 'unpack' these files?

    Since there were two steps involved in creating these files, there two conceptual steps involved in
    extracting them. Before we give you the easy way to extract these files, let's follow the conceptual
    steps first. For our purposes, we'll assume that the file in question resides in your current working
    directory. The first step is to decompress the file. If the file ends in .tgz, you'll need to rename it to a
    .tar.gz file so that the gzip utility will recognize it:

      mv file_in_question.tgz file_in_question.tar.gz

    Then, depending on whether the file is a ".tar.gz" (.tgz) or ".tar.bz2" file, you'll need to use one of the
    following commands:

      gunzip file_in_question.tar.gz
      bunzip2 file_in_question.tar.bz2

    Executing the proper command will decompress the file in question and remove the compressor's
    extension from the file. The resulting file is named only with the .tar extension:

      file_in_question.tar

    To extract all of the included files from this single file, make a new directory to hold the extracted files,
    go to that directory, and then use the tar command with the x (extract), v (verbose), and f (file is
    specified next) options:

      mkdir downloaded
      cd downloaded
      tar xvf ../file_in_question.tar

    Most .tar files will create another directory inside the directory you just created and extract their files
    there. Some archives won't, however, so you'll need to watch carefully to see whether the extracted
    files went into a subdirectory. Assuming that all of the above steps worked properly, your downloaded
    archive file has been completely extracted. If the .tar file created its own directory when it was
    extracted, you'll need to make it your current working directory before attempting to compile the
    software:

      cd file_in_question_directory

    More 'unpacking' Notes

    The Linux tar utility is actually more advanced that the standard Unix tar utility, and includes a significant
    shortcut for extracting .tar.gz files. Instead of first using the gunzip utility to extract the file, you can have
    the Linux tar utility do it for you by also specifying the z (zip/unzip) option:

      tar xzvf file_in_question.tgz
      tar xzvf another_file_in_question.tar.gz

    There are also a files out there (mainly from older Unix systems) which end with the .taz, .tar.Z or .tar.z
    extensions. These files were created with the tar utility and the standard Unix compressor called (oddly
    enough) "compress". The gzip utility in Linux also understands files compressed with the Unix
    compress utility, so the following commands will all work fine:

      mv file_1.taz file_1.tar.z; gunzip file_1.tar.z
      gunzip file_2.tar.Z
      tar xzvf file_3.tar.Z
      tar xzvf file_4.taz

    Note that this speedy method of unpacking (using the 'z' option) will not work with bzip or bzip2 files,
    which end in .tar.bz or .tar.bz2. Such files will still need to be unpacked using the two-step process
    given earlier.

    Getting Ready to Compile

    Now that you're sitting in a directory full of files, it's time to generate software. This is done in two steps:
    first, the source code must be "configured" for your system. Then, the make command must be run to
    generate the actual machine code. Most extracted source archives these days allow for some sort of
    "automatic" configuration. These will be the easiest for Linux newbies to compile and use. There are two
    common types of automatic configuration: those using a GNU "configure" script and those using
    something called an Imakefile. The "configure" script is preferable, so start by looking for a file called
    "configure" in the directory containing the extracted files. If such a file exists, run it now:

      ./configure

    You'll see a long list of messages as the script attempts to analyze the makeup of your system.
    Assuming the script exits without errors, you're ready to compile. If, on the other hand, no "configure"
    script exists in the directory, but instead a file called "Imakefile" is present, you'll need to run the
    "xmkmf" program to configure the source code:

      xmkmf

    If this command finishes without errors, you may be ready to compile. Whichever method you use
    (configure or xmkmf), if the process fails in a flurry of error messages, you're stuck for all intents and
    purposes. Experts are often able to fix these sorts of problems, but the solutions would be Greek at
    best to newbies, and methods for solving such problems are outside the scope of this article. You're
    better off trying to find and install precompiled package.

    Compiling

    The compilation process is generally simple, but time consuming. Type one word in the directory
    containing the extracted files:

      make

    A flurry of activity should follow; much of it will seem like technical gibberish. Linux is now "compiling"
    the source code into binary code (a program) which can be executed. On very fast computers, very
    small programs may compile in a matter of seconds. At the other extreme, on relatively slow
    computers, large programs have been known to need days or even weeks of compilation time. The
    process will proceed without need of your attention.

    There may be a number of warnings during the compilation process. These can usually be safely
    ignored. If at any point the process abruptly aborts and displays a cryptic error message in any of a
    number of forms, you're stuck:

      make: *** [init/main.o] Error 1

    In cases like this, you should look for documentation such as plain text INSTALL or README files
    included with the source code. These can be cryptic, but may contain additional instructions for
    compilation. If you find and study such files, follow all instructions, and the compilation still fails, or if you
    can't find any such INSTALL or README documentation files, you're again better off trying to find and
    install a precompiled package for your system.

    When the process is complete, you will find yourself at the command line once again. Generally, if no
    error messages (other than warnings) are obviously present, the compilation was successful, though
    there can be exceptions to this rule. For our purposes, we'll assume that if you've got this far, your
    download has compiled without problems.

    Installing

    Most current source code allows for automatic installation using the make command a second time,
    making the "install" item:

      make install

    Note that you will generally need to be root to make the install, since it will try to install program binaries
    into global binaries directories such as /bin, /usr/bin, or /usr/local/bin. With any luck, after entering this
    command, you'll see the install process commence, as files are copied to various places throughout
    your system. If you're interested in keeping a record of which files were installed where (in case, for
    example, you wanted to "uninstall" this program later), you can keep a record of the install process in
    some file on your system:

      make install 2>&1 | tee /path/to/saved_log

    The created file simply contains text duplicating exactly what you saw on your screen (see the tee
    manual page for more information). If the install method above fails with an error message, but the
    compilation (completed earlier) was successful, you'll need to consult any included documentation in
    INSTALL or README files to see how to install the binaries and other program files on your system.

    After the install has successfully run its course or you have consulted the necessary documentation
    and followed included instructions, you may be able to execute the program just by typing its name at a
    command prompt:

      myprog

    If this isn't the case, review the install output and try to determine (using common sense and a little bit of
    intelligent guesswork) if the file is going into either /usr/local/bin or /opt/bin. If it is, you may need to add
    the directory to your PATH environment variable:

      PATH=$PATH:/usr/local/bin:/opt/bin

    Now, try to execute the program again:

      myprog

    If this doesn't work either, there may be additional complications. Consult the documentation which was
    included with the source code. Such documentation usually resides in files called README, INSTALL,
    or in a directory called doc, and should give you more hints about using the program (and starting it)
    once it's been installed.

    If adding directories to your PATH variable does work, you may want to make the appropriate changes
    to the variable permanent by adding lines to your .bashrc and .profile scripts:

      echo "PATH=$PATH:/usr/local/bin:/opt/bin" >>.bash_profile
      echo "PATH=$PATH:/usr/local/bin:/opt/bin" >>.bashrc

    Seems complicated!

    If this seems like a lot of work, confusion, and generality to go through to get a program running, it is!
    There are many pitfalls, and some programs just won't compile and install on a specific system without
    a great deal of extra work or even extra code. For most users, it makes much more sense to download
    a precompiled package if such a package is available.

    If such a package isn't available, however, it's sometimes necessary to download the code and compile
    it yourself. Don't get discouraged if your first few tries are dismal failures: it takes every newbie a while
    to get the hang of compiling and installing from source code. In the end, however, it will become easier
    as you become more experienced, and you may be able to return to some of your early failures and

    performs two functions: it allows many different files to be rolled into one single file for convenience, and
    it compresses this single file for faster downloading (or space savings). In Linux, these two tasks are
    performed by two separate commands: the tar command and the gzip command.

    Files which end in the .tar.gz extension are the result of a two-step process. First, many files are rolled
    into a single file using the tar (Tape ARchive) program, giving the resulting file a .tar extension. Next, the
    .tar file is compressed using the gzip program, adding an additional .gz extension to the end of the file.
    Sometimes .tar.gz is abbreviated as .tgz to allow such files to exist on MS-DOS systems, which allow
    only for a single, three character filename extension. In any case, .tar.gz or .tgz files are known as
    "tarred-gzipped" files.

    Files which end in the .tar.bz2 extension are similar to the .tar.gz files in that they are multiple files which
    have been run through tar into a single file and then compressed. Files ending in .tar.bz2, however, are
    compressed with the bzip2 compressor rather than the gzip compressor. To see if your Linux system
    contains a bzip2 compressor/decompressor, type "bunzip2 --help" at the command line. If you see a
    25-line usage message, the utility is present on your system.

    How do I 'unpack' these files?

    Since there were two steps involved in creating these files, there two conceptual steps involved in
    extracting them. Before we give you the easy way to extract these files, let's follow the conceptual
    steps first. For our purposes, we'll assume that the file in question resides in your current working
    directory. The first step is to decompress the file. If the file ends in .tgz, you'll need to rename it to a
    .tar.gz file so that the gzip utility will recognize it:

      mv file_in_question.tgz file_in_question.tar.gz

    Then, depending on whether the file is a ".tar.gz" (.tgz) or ".tar.bz2" file, you'll need to use one of the
    following commands:

      gunzip file_in_question.tar.gz
      bunzip2 file_in_question.tar.bz2

    Executing the proper command will decompress the file in question and remove the compressor's
    extension from the file. The resulting file is named only with the .tar extension:

      file_in_question.tar

    To extract all of the included files from this single file, make a new directory to hold the extracted files,
    go to that directory, and then use the tar command with the x (extract), v (verbose), and f (file is
    specified next) options:

      mkdir downloaded
      cd downloaded
      tar xvf ../file_in_question.tar

    Most .tar files will create another directory inside the directory you just created and extract their files
    there. Some archives won't, however, so you'll need to watch carefully to see whether the extracted
    files went into a subdirectory. Assuming that all of the above steps worked properly, your downloaded
    archive file has been completely extracted. If the .tar file created its own directory when it was
    extracted, you'll need to make it your current working directory before attempting to compile the
    software:

      cd file_in_question_directory

    More 'unpacking' Notes

    The Linux tar utility is actually more advanced that the standard Unix tar utility, and includes a significant
    shortcut for extracting .tar.gz files. Instead of first using the gunzip utility to extract the file, you can have
    the Linux tar utility do it for you by also specifying the z (zip/unzip) option:

      tar xzvf file_in_question.tgz
      tar xzvf another_file_in_question.tar.gz

    There are also a files out there (mainly from older Unix systems) which end with the .taz, .tar.Z or .tar.z
    extensions. These files were created with the tar utility and the standard Unix compressor called (oddly
    enough) "compress". The gzip utility in Linux also understands files compressed with the Unix
    compress utility, so the following commands will all work fine:

      mv file_1.taz file_1.tar.z; gunzip file_1.tar.z
      gunzip file_2.tar.Z
      tar xzvf file_3.tar.Z
      tar xzvf file_4.taz

    Note that this speedy method of unpacking (using the 'z' option) will not work with bzip or bzip2 files,
    which end in .tar.bz or .tar.bz2. Such files will still need to be unpacked using the two-step process
    given earlier.

    Getting Ready to Compile

    Now that you're sitting in a directory full of files, it's time to generate software. This is done in two steps:
    first, the source code must be "configured" for your system. Then, the make command must be run to
    generate the actual machine code. Most extracted source archives these days allow for some sort of
    "automatic" configuration. These will be the easiest for Linux newbies to compile and use. There are two
    common types of automatic configuration: those using a GNU "configure" script and those using
    something called an Imakefile. The "configure" script is preferable, so start by looking for a file called
    "configure" in the directory containing the extracted files. If such a file exists, run it now:

      ./configure

    You'll see a long list of messages as the script attempts to analyze the makeup of your system.
    Assuming the script exits without errors, you're ready to compile. If, on the other hand, no "configure"
    script exists in the directory, but instead a file called "Imakefile" is present, you'll need to run the
    "xmkmf" program to configure the source code:

      xmkmf

    If this command finishes without errors, you may be ready to compile. Whichever method you use
    (configure or xmkmf), if the process fails in a flurry of error messages, you're stuck for all intents and
    purposes. Experts are often able to fix these sorts of problems, but the solutions would be Greek at
    best to newbies, and methods for solving such problems are outside the scope of this article. You're
    better off trying to find and install precompiled package.

    Compiling

    The compilation process is generally simple, but time consuming. Type one word in the directory
    containing the extracted files:

      make

    A flurry of activity should follow; much of it will seem like technical gibberish. Linux is now "compiling"
    the source code into binary code (a program) which can be executed. On very fast computers, very
    small programs may compile in a matter of seconds. At the other extreme, on relatively slow
    computers, large programs have been known to need days or even weeks of compilation time. The
    process will proceed without need of your attention.

    There may be a number of warnings during the compilation process. These can usually be safely
    ignored. If at any point the process abruptly aborts and displays a cryptic error message in any of a
    number of forms, you're stuck:

      make: *** [init/main.o] Error 1

    In cases like this, you should look for documentation such as plain text INSTALL or README files
    included with the source code. These can be cryptic, but may contain additional instructions for
    compilation. If you find and study such files, follow all instructions, and the compilation still fails, or if you
    can't find any such INSTALL or README documentation files, you're again better off trying to find and
    install a precompiled package for your system.

    When the process is complete, you will find yourself at the command line once again. Generally, if no
    error messages (other than warnings) are obviously present, the compilation was successful, though
    there can be exceptions to this rule. For our purposes, we'll assume that if you've got this far, your
    download has compiled without problems.

    Installing

    Most current source code allows for automatic installation using the make command a second time,
    making the "install" item:

      make install

    Note that you will generally need to be root to make the install, since it will try to install program binaries
    into global binaries directories such as /bin, /usr/bin, or /usr/local/bin. With any luck, after entering this
    command, you'll see the install process commence, as files are copied to various places throughout
    your system. If you're interested in keeping a record of which files were installed where (in case, for
    example, you wanted to "uninstall" this program later), you can keep a record of the install process in
    some file on your system:

      make install 2>&1 | tee /path/to/saved_log

    The created file simply contains text duplicating exactly what you saw on your screen (see the tee
    manual page for more information). If the install method above fails with an error message, but the
    compilation (completed earlier) was successful, you'll need to consult any included documentation in
    INSTALL or README files to see how to install the binaries and other program files on your system.

    After the install has successfully run its course or you have consulted the necessary documentation
    and followed included instructions, you may be able to execute the program just by typing its name at a
    command prompt:

      myprog

    If this isn't the case, review the install output and try to determine (using common sense and a little bit of
    intelligent guesswork) if the file is going into either /usr/local/bin or /opt/bin. If it is, you may need to add
    the directory to your PATH environment variable:

      PATH=$PATH:/usr/local/bin:/opt/bin

    Now, try to execute the program again:

      myprog

    If this doesn't work either, there may be additional complications. Consult the documentation which was
    included with the source code. Such documentation usually resides in files called README, INSTALL,
    or in a directory called doc, and should give you more hints about using the program (and starting it)
    once it's been installed.

    If adding directories to your PATH variable does work, you may want to make the appropriate changes
    to the variable permanent by adding lines to your .bashrc and .profile scripts:

      echo "PATH=$PATH:/usr/local/bin:/opt/bin" >>.bash_profile
      echo "PATH=$PATH:/usr/local/bin:/opt/bin" >>.bashrc

    Seems complicated!

    If this seems like a lot of work, confusion, and generality to go through to get a program running, it is!
    There are many pitfalls, and some programs just won't compile and install on a specific system without
    a great deal of extra work or even extra code. For most users, it makes much more sense to download
    a precompiled package if such a package is available.

    If such a package isn't available, however, it's sometimes necessary to download the code and compile
    it yourself. Don't get discouraged if your first few tries are dismal failures: it takes every newbie a while
    to get the hang of compiling and installing from source code. In the end, however, it will become easier
    as you become more experienced, and you may be able to return to some of your early failures and
    finally get them to install and run, too.
